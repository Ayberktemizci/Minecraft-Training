🧩 Oasis de la Lógica

Juegos, acertijos y los verdaderos fundamentos de la lógica de programación: los estudiantes crean su primer mini-juego interactivo utilizando bloques de eventos, variables y condicionales en MakeCode, transformando a Minecraft en un motor de juego programable.

Objetivos del Módulo 3: Comprender la programación orientada a eventos (Al caminar el jugador, Al romper un bloque) · Crear y modificar variables como "contenedores de puntuación" · Aplicar la lógica condicional Si–Entonces y Si–Entonces–Sino · Combinar Eventos + Variables + Condicionales para programar un mini-juego completamente jugable y funcional.
1 · Resumen del Módulo — El Giro hacia la Programación

Por qué el Módulo 3 es el punto de inflexión conceptual

Los Módulos 1 y 2 utilizaron MakeCode para automatizar construcciones físicas: el programa se ejecutaba una vez y producía un resultado estático. El Módulo 3 introduce la programación reactiva e interactiva: código que espera a que ocurra un suceso en el mundo para responder. Este es el paradigma fundamental de casi todo el software del mundo real, desde aplicaciones móviles hasta motores de videojuegos. Los estudiantes ya no solo construyen estructuras; ahora diseñan experiencias.

Los tres conceptos introducidos (eventos, variables y condicionales) son suficientes para desarrollar cualquier juego simple. Al finalizar este módulo, los estudiantes habrán escrito su primer programa interactivo completo.

2 · Programación Orientada a Eventos

¿Qué es un evento?

Un evento es una señal que indica que algo ha sucedido en el mundo del juego. Un bloque de evento en MakeCode registra un receptor o listener: el código dentro del bloque se ejecuta *única y exactamente* cuando ese evento se dispara. El código no corre continuamente, sino que espera y responde.

al caminar el jugador { ← bloque de evento: se activa con cada paso del jugador
  cambiar score por 1 ← este código se ejecuta cada vez que el evento se dispara
} ← fuera de este bloque = NO se activa al caminar

Eventos clave del Módulo 3 que el tutor debe explicar:

  • Al caminar el jugador: Se dispara una vez por cada paso que da el jugador. Útil para contadores de pasos, puntuación basada en distancia y mini-juegos de caminata.
  • Al romper un bloque: Se dispara cuando el jugador mina o destruye un tipo de bloque específico. Útil para mini-juegos de recolección de recursos y búsquedas del tesoro.
  • Al comando de chat [palabra]: Se dispara cuando el jugador escribe la palabra clave en el chat. Útil para activadores manuales del juego, códigos de trampa o funciones de reinicio.
  • Al saltar el jugador: Se dispara una vez por cada salto. Útil para puntuación en parkour y desafíos de salto.
⚠️
Error Crítico: Código posicionado fuera del bloque de evento
Este es el error de codificación más común en el Módulo 3. Los estudiantes colocan el bloque "cambiar variable por 1" FUERA del contenedor del bloque de evento. Cuando esto ocurre, la modificación de la variable se ejecuta una sola vez al cargar el programa (cuando se ejecuta el comando /code), y no cada vez que se dispara el evento.
INCORRECTO — la puntuación cambia solo al inicio del programa, no al caminar:
al caminar el jugador { ← bloque de evento vacío
}
cambiar score por 1 ← FUERA del bloque — se ejecuta UNA VEZ y nunca más
CORRECTO — la puntuación cambia en cada paso:
al caminar el jugador {
  cambiar score por 1 ← DENTRO del bloque
}
Solución pedagógica: Antes de que los estudiantes comiencen a programar, pídeles que dibujen un rectángulo en papel y ubiquen físicamente cada instrucción dentro o fuera de él. Visualizar el límite del bloque previene este error de manera drástica.
3 · Variables — Contenedores para Datos Dinámicos

¿Qué es una variable?

Una variable es un espacio de almacenamiento nombrado en la memoria que guarda un valor (un número, una cadena de texto o un valor booleano verdadero/falso). Este valor puede ser leído, escrito y actualizado. Para el Módulo 3, todas las variables serán numéricas: guardarán puntuaciones, contadores y temporizadores.

asignar score = 0 ← crea la variable y le asigna un valor inicial
cambiar score por +1 ← suma 1 al valor que se encuentra almacenado actualmente
cambiar score por -5 ← resta 5 al valor actual
asignar score = score + 10 ← lee el valor actual, le suma 10 y guarda el resultado
mostrar score ← muestra el valor actual en el chat (para pruebas)

Alcance de las variables en MakeCode: Todas las variables son globales en el entorno de Minecraft Education de MakeCode: cualquier bloque de evento puede leer y modificar cualquier variable. No existen las variables locales. Esto significa que una vez definida una variable, todo el programa tiene acceso a ella.

Convención de nomenclatura para enseñar: Utiliza nombres descriptivos (score, pasos, gemas), no letras sueltas (x, a). Los nombres descriptivos permiten que los niños de 8 años depuren sus programas eficazmente, ya que pueden leer su propio código e interpretar qué hace.

4 · Condicionales — Lógica Si–Entonces

Bloques Si–Entonces y Si–Entonces–Sino

Un condicional permite que el programa tome decisiones: si una condición es verdadera, ejecuta el bloque A; de lo contrario, ejecuta el bloque B (la rama "sino"). Sin condicionales, el programa carece de capacidad de decisión y siempre hará lo mismo sin importar el estado del juego.

si score ≥ 10 entonces {
  mostrar "¡Ganaste! Puntuación final: " + score
  fuegos artificiales en la posición del jugador ← activa animación de victoria
  asignar score = 0 ← reinicia para una nueva partida
}

si score ≥ 10 entonces { ← Variante Si–Entonces–Sino:
  mostrar "¡Ganaste!"
} sino {
  mostrar "¡Sigue adelante! Te faltan " + (10 - score) + " pasos"
}

Operadores de comparación que los estudiantes deben dominar:

Igualdad y Desigualdad

= igual a · ≠ diferente de
> mayor que · < menor que
≥ mayor o igual que · ≤ menor o igual que

Lógica Booleana

Y (AND) — ambas condiciones deben ser verdaderas
O (OR) — al menos una condición es verdadera
NO (NOT) — invierte el valor de la condición
Ejemplo: score ≥ 10 Y lives > 0

5 · Creación del Mini-Juego — Ejemplo Completo

El Mini-Juego "Carrera hasta 10" — Especificación Técnica Completa

Este es el mini-juego mínimo viable que integra los tres conceptos del Módulo 3 en un solo programa.

// Bloque de configuración — se ejecuta una vez al iniciar el programa
al iniciar {
  asignar score = 0
  mostrar "¡Juego iniciado! Camina para ganar puntos. ¡Llega a 10 para ganar!"
}

// Bloque de evento — se activa con cada paso
al caminar el jugador {
  cambiar score por 1
  si score ≥ 10 entonces {
    mostrar "¡GANASTE! Puntuación final: " + score
    fuegos artificiales en la posición del jugador
    asignar score = 0 // reinicia para volver a jugar
  }
}

Extensiones que los estudiantes pueden agregar de forma independiente (Tareas Extra):

  • Añadir un sistema de vidas: Define lives = 3 al iniciar. Si el jugador cae por debajo de Y=60, resta 1 vida. Si las vidas llegan a 0, termina el juego.
  • Añadir un temporizador: Utiliza una variable de cuenta regresiva. Disminúyela mediante un evento repetitivo. Si el temporizador llega a 0 antes de que la puntuación llegue a 10, muestra el mensaje "¡Tiempo agotado!".
  • Añadir potenciadores (Power-ups): Al romper un bloque (como Diamante) → cambia la puntuación por +5 como bonificación.
6 · Catálogo de Resolución de Problemas — Módulo 3 (5 Escenarios)
Error / SíntomaCausa RaízSolución ExactaGuión de Prevención
La puntuación empieza en 1 inmediatamente al ejecutar /code, antes de que el jugador camine El bloque "asignar score = 0" está colocado dentro del evento "Al caminar el jugador" en lugar de un bloque "Al iniciar" o de configuración independiente. Mueve el bloque "asignar score = 0" fuera del evento de caminar hacia un bloque "Al iniciar" (o directamente al lienzo si no se usa evento de inicio). El evento de caminar solo debe contener el incremento y la condición. "El código de configuración va en 'Al iniciar'. El código de juego va en los eventos. Nunca mezcles la inicialización con las respuestas."
La variable score nunca pasa de 1; se reinicia en cada paso en lugar de acumularse El bloque "asignar score = 0" se encuentra dentro del evento "Al caminar el jugador", por lo que la puntuación vuelve a 0 en cada paso antes de incrementarse a 1. Elimina el bloque "asignar score = 0" de adentro del bloque de caminar. La inicialización pertenece únicamente a 'Al iniciar'. El bloque de caminar solo debe incluir el incremento: "cambiar score por 1". "Lee tu código en voz alta. 'En cada paso: asigno score a 0, luego sumo 1.' ¿Tiene sentido? ¿Qué número tendrá siempre la puntuación?"
La condición Si–Entonces nunca se activa, incluso cuando la puntuación supera claramente el objetivo El operador de comparación es incorrecto (se usa = en lugar de ≥); O el nombre de la variable en la condición está escrito de forma distinta a la variable incrementada (MakeCode distingue entre mayúsculas y minúsculas). Verifica el operador: si el objetivo es alcanzar 10 o más, la condición debe ser score ≥ 10, no score = 10 (que solo se activa exactamente a los 10). Revisa la escritura: "Score" y "score" son variables totalmente distintas. "Usa ≥ (mayor o igual) para las condiciones de victoria, no = (igual a); el jugador podría saltarse el número exacto al dar un paso."
La condición de victoria se activa en cada paso tras alcanzar los 10 puntos; los fuegos artificiales suenan repetidamente La variable score no se está reiniciando a 0 dentro del bloque condicional de victoria; al ganar, la puntuación sigue siendo ≥ 10, por lo que cada paso posterior vuelve a disparar la condición. Agrega el bloque "asignar score = 0" como la última línea dentro del condicional de victoria. Alternativamente, define una variable booleana "gameActive": cámbiala a falso al ganar y envuelve el evento de caminar en un "si gameActive = verdadero entonces". "Tras ganar, ¿qué debería pasar con la puntuación? O se reinicia a 0 o se apaga el juego. Si no la reinicias, la victoria se activará para siempre."
El estudiante no entiende por qué "mostrar score" enseña solo un número cuando quería mostrar texto + número (ej. "Puntuación: 5") Combinación incorrecta de bloques: usar "mostrar" solo con la variable genera el número; para combinar texto y variable se requiere un bloque de unión ("join") o el operador de concatenación. Muestra a los alumnos el bloque de unir texto en MakeCode: [mostrar [unir ["Puntuación: "] [score]]]. Explica: "Mostrar sirve para enseñar un elemento; para combinar texto y números, debes unirlos primero en una sola cadena de texto". "'Mostrar' solo puede contener un elemento. Usa el bloque de unir para juntar el texto y el número antes de entregárselo a Mostrar."
7 · Consejos Pro del Tutor y Analogías del Mundo Real

💡 Variables = "La Puntuación de los Videojuegos" (Gancho Principal del Módulo 3)
"Piensa en cada videojuego que has jugado. Mario tiene un contador de monedas. El FIFA tiene un marcador de goles. Minecraft tiene una barra de experiencia. Todos esos elementos son variables: una caja con un nombre que guarda un número, y ese número cambia mientras juegas. Hoy vas a programar el código detrás de tu propio contador de puntos".

💡 Eventos = "El Sistema de Notificaciones"
"Tu teléfono no lee todos los mensajes en el instante en que se escriben. Espera en silencio, y cuando llega un mensaje, te avisa. Esa notificación ES un evento. Los bloques de eventos en MakeCode funcionan exactamente igual: duermen hasta que algo pasa en Minecraft, y en ese momento se despiertan para ejecutar su código".

💡 Si–Entonces = "El Termostato"
"Un termostato no enciende la calefacción todo el tiempo; revisa constantemente: '¿La temperatura es menor a 20°C?' Si es verdadero, enciende el calentador. Si es falso, se queda apagado. Eso es exactamente lo que hace un bloque Si–Entonces: evalúa una condición y actúa únicamente cuando esa condición es verdadera".

💡 El truco de la "Visualización de Límites de Bloques" para evitar errores externos
Antes de que los estudiantes programen en el Módulo 3, pídeles que dibujen un rectángulo en una hoja de papel. Rotúlenlo como "Al caminar el jugador". Luego pregunta: "¿Cuál de tus instrucciones se coloca DENTRO de este rectángulo y cuál se queda afuera?". Los alumnos que dibujan físicamente el límite antes de programar cometen este error un 70% menos en comparación con quienes codifican directamente sin planificar.
