Module 5 of 5 — Final Module
🏰 Creator's Castle

End-to-end project creation — students apply every skill from Modules 1–4 to design, build, and publicly present their own Minecraft game or automated world. This is the capstone module and the direct preparation for the DST readiness assessment.

Module 5 Objectives: Run a structured project brainstorm and Blueprint Phase · Assign clear team roles for collaborative builds · Create and use custom MakeCode Functions to eliminate code repetition · Execute the Build → Debug → Decorate loop · Host a professional-style "Game Studio Launch" presentation for parents and peers.
1 · Module Overview — The Capstone

🎓 Why Module 5 Is Different from All Others

Every previous module taught a specific technical skill set in a guided, tutor-led structure. Module 5 deliberately inverts this: students are now the directors, and the tutor is the production advisor. The technical skills introduced (custom Functions) are deliberately simpler than Module 4's automation systems — because the cognitive challenge is now creative and managerial, not technical.

The tutor's primary job in Module 5 is to maintain productive forward momentum: prevent decision paralysis during the brainstorm, enforce the Blueprint Phase discipline, and coach students through the build loop without taking over.

Skills Catalogue — What Students Must Bring Into Module 5

Module 5 assumes mastery of all prior content. Before beginning, do a rapid 5-minute verbal check:

  • From M1: Agent control, basic Redstone, coordinate navigation
  • From M2: Crafting, /clone, cooperative building with defined roles
  • From M3: Events, variables, conditionals, mini-game logic
  • From M4: Hopper sorting, automated farming pipeline, nested loops

Students who are missing skills from any prior module must be identified and given targeted support before Module 5 progresses past the Blueprint Phase — a weak foundation will produce a failing final project.

2 · The Blueprint Phase — Plan Before Placing a Single Block
⚠️
The #1 Module 5 Failure Mode: Building Without Planning
Teams that skip the Blueprint Phase will argue about what to build for 15–20 minutes of the session. This is a session-destroying failure because once students are mid-build and disagree about direction, there is no recovery within a 60-minute window. The Blueprint Phase is non-negotiable and must be completed before any block is placed.

Structured Brainstorm Protocol — 10 Minutes Maximum

Run a timed brainstorm with strict facilitation:

  1. Silent ideation (3 min): Each student types 1–2 project ideas in the chat simultaneously. No verbal discussion yet. Ideas must include: (a) a theme/world type, (b) one interactive or automated feature.
  2. Idea summary (2 min): Tutor reads out all ideas aloud without editorialising.
  3. Team vote (2 min): Students vote in chat using numbers (1, 2, 3…). Majority wins. If tied, tutor breaks the tie. No re-debates after the vote is closed.
  4. Blueprint commitment (3 min): All students type "I agree with this plan: [project name]" in chat. This creates social accountability and dramatically reduces mid-build direction changes.

The Blueprint Document — What Must Be Agreed Before Building

After the vote, the team must agree on four items, written in the chat for all to see:

1. Project Theme

What is the world's setting or purpose? (e.g., "A medieval castle with a working drawbridge and treasure room," "A space station with automated resource generation," "A parkour obstacle course that tracks scores using variables.")

2. Signature Feature

The one interactive or automated system the team must build — the technical centrepiece. This must use at least one skill from Modules 1–4. Without a signature feature, the project is just a static build, not a game or automation system.

3. Team Roles

Every student has one primary responsibility: Architect (structural design), Programmer (MakeCode logic), Decorator (visual detail), and Demonstrator (handles the final presentation). In groups of 2–3, roles are combined.

4. Success Criteria

What does the completed project look like? Define a minimum viable version: "The drawbridge opens when a player activates the pressure plate AND the treasure chest is accessible behind it." This prevents scope creep and gives students a clear finishing line.

3 · Custom MakeCode Functions — Technical Deep-Dive

What Is a Function?

A Function is a named, reusable block of code. Once defined, it can be called (executed) from anywhere in the programme with a single command block. Functions eliminate repetition: instead of copying the same 5-block sequence 4 times across different event handlers, write it once as a Function and call it 4 times.

// Define the function once:
function celebrate() {
  fireworks at position player
  say "Level Complete!"
  music play victory
}

// Call it from multiple places without re-writing it:
on player walk {
  if score ≥ 10 then {
    celebrate() ← single call replaces 3 blocks
  }
}

on block broken {
  if blockscore ≥ 5 then {
    celebrate() ← same 3-block sequence, zero duplication
  }
}

Why Functions matter for 8-year-olds: When students copy code blocks, they introduce inconsistency — if they change their mind about the celebrate sequence, they must find and update every copy. With a Function, they update one definition and the change applies everywhere instantly. This is their first encounter with the programming principle of DRY (Don't Repeat Yourself).

When to Teach Functions in the Session

Introduce Functions at the point when a student is about to copy a code block sequence for the second time. This is the "teachable moment": the student can feel the problem (repetition) before learning the solution (Function). Never introduce Functions as a theoretical concept before students have experienced the pain of code duplication — the motivation must come from their own frustration.

"I see you're about to copy that celebrate sequence again. Before you do — there's a smarter way. What if you could write it once and use it everywhere? That's called a Function. Let me show you."
4 · The Build → Debug → Decorate Loop

The Three-Phase Build Loop

Every project session after the Blueprint Phase should follow this cycle. Teams must complete one full cycle before adding new features.

Phase 1
🔨 Build — Minimum Viable Version First
Build only the success criteria defined in the Blueprint. No decorations, no extra features, no scope additions. The goal of Phase 1 is a working, ugly prototype. Tutors must actively prevent "decoration creep" — students who begin placing decorative blocks before the core feature works are wasting time and may demotivate themselves when the feature later fails to work under the decorative surface.
Phase 2
🔍 Debug — Make It Work Before Making It Beautiful
Test every interactive and automated feature from the success criteria. Use the isolation testing protocol from Module 4: test each system independently, then test the integrated whole. Every bug must be resolved before Phase 3 begins. This is the tutor's most active phase: circulate continuously, ask "what should happen here?" not "what's wrong?" — students diagnose, tutors guide.
Phase 3
🎨 Decorate — Make It Reflect the Team's Creative Vision
Only after debug confirmation: add banners, custom builds, colour schemes, signage, and atmospheric details. Assign decoration work to students who finished debugging early — never let a student be idle. The Decorator role has ownership of Phase 3, but all team members contribute. Phase 3 is also the time to add any "bonus features" from the Extra Task list.
5 · The Game Studio Launch — Professional Presentation
💡
Why Call It a "Game Studio Launch" Not a "Show and Tell"
Language shapes behaviour. "Show and tell" is something primary school students do with toys. "Game Studio Launch" is what Mojang, Nintendo, and Valve do. The language upgrade boosts confidence, raises the perceived stakes, and produces better presentations — students explain their work with professional pride rather than playground casualness.

Presentation Structure — Each Team Follows This Exact Format

The Demonstrator student leads; other team members support. Total time: 5 minutes per team.

  1. Project Name & Concept (30 seconds): "Our project is called [name]. The idea is [concept in one sentence]."
  2. Signature Feature Live Demo (2 minutes): Demonstrate the core interactive or automated system working in real-time. The tutor screen-shares the Demonstrator's view. All other students watch.
  3. Technical Explanation (1 minute): "This works because [technical mechanism]. We used [Module X skill] from our training." Students must name the specific MakeCode or Redstone concept, not just say "we coded it."
  4. What We Would Add Next (30 seconds): "If we had one more session, we would add [feature]." This teaches iterative thinking and prevents perfectionism paralysis.
  5. Q&A (1 minute): Other students and parents ask one question each. The tutor facilitates.

Optional parent participation: Invite parents to the final session using a pre-written invitation (available in the Kodland admin panel). Parent presence significantly elevates student motivation and performance quality. Provide a 2-sentence briefing to parents before the session starts so they know what to expect and can ask informed questions.

Evaluation Criteria — What the Tutor Assesses During Presentations

✅ Strong Presentation Indicators

Student can explain the technical mechanism without prompting · Live demo works without crashing · Team uses at least one skill from two or more previous modules · Student speaks in complete sentences · Student answers a question they didn't prepare for.

❌ Weak Presentation Indicators

Live demo crashes mid-way (debug was incomplete) · Student cannot explain how the feature works · Project has no interactive or automated element · Student reads from notes the entire time · Team relied entirely on tutor for all technical decisions.

6 · Troubleshooting Catalog — Module 5 (5 Scenarios)
Mistake / SymptomRoot CauseExact FixPrevention Script
Team spends 20+ minutes arguing about what to build — no consensus by minute 20 Blueprint Phase was not started with a structured format; discussion was open-ended and unstructured, allowing indefinite back-and-forth without forcing a decision Immediately switch to the formal structured brainstorm: silent ideation (3 min), read ideas aloud, vote by number in chat, tutor breaks ties. Close the vote at exactly minute 15 — no extensions. Announce: "We've voted. We're building [X]. This decision is final. Blueprint document time." "If there's no consensus by minute 10 of brainstorming, call an immediate vote and use tutor tie-break authority. Do not allow debate to continue."
A student's custom Function is defined but calling it produces no visible result The Function block was created but the "Call Function [name]" block was not placed inside any event handler. A Function definition alone never executes — it must be called. Show the student: "Your Function is defined — it's like a recipe in a cookbook. But no one told the chef to cook it. Add a 'Call Function [name]' block inside the On Chat Command or On Player Walk event." Place the Call block inside the relevant event and test. "A Function definition is a recipe. A Function call is 'cook this recipe now'. You need both — define AND call."
Team began decorating before the core feature was tested — the decoration is now built over a broken circuit, making it impossible to access the Redstone without destroying the build Decoration Phase started before Debug Phase was complete — the Build → Debug → Decorate order was violated Do not try to work around the decorations. Address the root cause immediately: use /clone to back up the decorated build to a spare area, then access and fix the broken circuit in the original location. Reinforce: "We have to pause decorating, fix the circuit, then resume. This is why Debug comes before Decorate." "No decoration until every feature works. We demo the working prototype first, then we make it beautiful. Beauty on a broken foundation is wasted effort."
Project scope expands beyond what's achievable in the remaining session time — team keeps adding features and nothing is finished No minimum viable version was defined in the Blueprint Phase; the success criteria were too vague ("build a really cool castle") rather than specific and testable Hard-stop the scope expansion. Return to the Blueprint: "What was our success criteria? [Read it back.] Is that done? [If yes: stop adding, move to Debug and Decorate.] [If no: stop adding new features, finish the success criteria first.]" Any additional features go onto a "Version 2" list that may or may not be built after the success criteria are met. "New feature ideas go on the 'Version 2' list. Nothing new gets built until the success criteria are done and debugged."
During the Game Studio Launch presentation, the live demo crashes or breaks mid-demonstration, causing student embarrassment in front of parents Debug Phase was not completed before the presentation; the feature was not tested in the exact conditions of the demo (e.g., from the spawn point, with other students also in the world) Run a full dress rehearsal 10 minutes before parents join. Have a backup plan: if the interactive feature fails, the Demonstrator should have a "recorded screenshot walkthrough" as a fallback — pre-taken screenshots showing the feature working correctly. The tutor calmly narrates: "Let me show you what it looked like when it worked perfectly earlier." "Always run a full dress rehearsal 10 minutes before parents join. Test the demo in exactly the same conditions as the live presentation."
7 · Tutor Pro-Tips & Real-World Analogies

💡 The "Game Studio Launch" Framing — Why It Produces Better Presentations
"You're not showing your homework — you're launching your game to the world. Mojang didn't show Minecraft to players and say 'sorry it's a bit rough.' They launched it, explained why it was special, and showed it working live. That's exactly what you're going to do today." This framing produces presentations that are 2–3× more confident and detailed than "show and tell" framing.

💡 Functions = The "Recipe Card" Analogy
"A professional chef doesn't write out the full pasta sauce recipe every time they need it — they write it once on a card and just say 'make the pasta sauce.' A MakeCode Function is your personal recipe card. Write the code once, name it, then call it anywhere by name. Change the recipe once and it updates everywhere."

💡 Build → Debug → Decorate = The "Construction Site" Analogy
"A real construction company builds the steel skeleton and roof first (Build). Then electrical inspectors test all the wiring while the walls are still open (Debug). Then the interior designers come in to make it beautiful (Decorate). Nobody paints the walls before the electrician checks the wiring — because painting over a fault doesn't fix it, it hides it."

💡 Blueprint Phase = The "No Arguing in the Air" Rule
A student architect once said: "Argument in the air is cheap. Argument after construction has started is expensive." Before any block is placed, disagreements cost nothing. After 30 minutes of building, direction changes are enormously costly. Run the Blueprint Phase like a contract: once signed (typed in chat), the plan is firm. Changes require a formal proposal with majority re-vote.

💡 Managing Scope Creep With the "Version 2" List
Every good idea that comes up after the Blueprint Phase gets written to a "Version 2 list" in the chat. This has two effects: (1) students feel their idea was valued and captured, not dismissed; (2) the production scope stays fixed. Students who see their idea on the Version 2 list are far more likely to stay focused on the current version than students whose idea was simply rejected.
