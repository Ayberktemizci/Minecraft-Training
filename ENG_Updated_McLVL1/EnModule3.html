1 · Module Overview — The Programming Shift

Why Module 3 Is the Conceptual Turning Point

Modules 1 and 2 used MakeCode to automate physical builds — the program ran once and produced a static result. Module 3 introduces reactive, interactive programming: code that waits for something to happen, then responds. This is the foundational paradigm of virtually all real-world software, from mobile apps to game engines. Students are no longer building machines — they are building experiences.

The three concepts introduced — events, variables, and conditionals — are sufficient to build any simple game. By the end of this module, students will have written their first complete interactive programme.

2 · Event-Driven Programming

What Is an Event?

An event is a signal that something has happened in the game world. An event block in MakeCode registers a listener: code inside the block executes only and exactly when that event fires. The code does not run continuously — it waits, then responds.

on player walk { ← event block: fires on every walk step
  change score by 1 ← this code runs each time the event fires
} ← outside this block = NOT triggered by walking

Key Module 3 events tutors must explain:

  • On Player Walk: Fires once per step taken by the player. Useful for step counters, distance-based scoring, walking mini-games.
  • On Block Broken: Fires when the player mines/destroys a specific block type. Useful for resource-gathering mini-games, treasure hunts.
  • On Chat Command [word]: Fires when the player types the word in chat. Useful for manual game triggers, cheat codes, restart functions.
  • On Player Jumped: Fires once per jump. Useful for parkour scoring, jumping challenges.
⚠️
Critical Mistake: Code Placed Outside the Event Block
This is the single most common coding error in Module 3. Students place the "change variable by 1" block OUTSIDE the event block's enclosure. When this happens, the variable change executes once at program load (when the /code command runs), not each time the event fires.
WRONG — score changes only at program start, not on walk:
on player walk { ← empty event block
}
change score by 1 ← OUTSIDE the block — runs ONCE, then never again
CORRECT — score changes on every walk step:
on player walk {
  change score by 1 ← INSIDE the block
}
Teaching fix: Before students code, ask them to draw a rectangle on paper and place every instruction either inside or outside it. Physically visualising the block boundary prevents this error.
3 · Variables — Containers for Changing Data

What Is a Variable?

A variable is a named storage location in memory that holds a value (a number, a string of text, or a boolean true/false). The value can be read, written, and updated. For Module 3, all variables will be numeric — they hold scores, counters, and timers.

set score = 0 ← creates the variable and gives it an initial value
change score by +1 ← adds 1 to whatever value is currently stored
change score by -5 ← subtracts 5 from the current value
set score = score + 10 ← reads current value, adds 10, writes result back
say score ← displays current value in chat (for testing)

Variable scope in MakeCode: All variables are global in the MakeCode Education environment — any event block can read and write any variable. There are no local variables. This means once a variable is set, every part of the programme has access to it.

Naming convention to teach: Use descriptive names (score, steps, gems), not single letters (x, a). Descriptive names make debugging possible for 8-year-olds — they can read their own code and understand what it does.

4 · Conditionals — If–Then Logic

If–Then and If–Then–Else Blocks

A conditional allows the programme to make a decision: if a condition is true, execute block A; otherwise, execute block B (the "else" branch). Without conditionals, the programme has no decision-making ability — it always does the same thing regardless of the game state.

if score ≥ 10 then {
  say "You Win! Final score: " + score
  fireworks at position player ← trigger win animation
  set score = 0 ← reset for new game
}

if score ≥ 10 then { ← If–Then–Else variant:
  say "You Win!"
} else {
  say "Keep going! " + (10 - score) + " steps to go"
}

Comparison operators students must know:

Equality & Inequality

= equals · ≠ not equal
> greater than · < less than
≥ greater or equal · ≤ less or equal

Boolean Logic

AND — both conditions must be true
OR — either condition is true
NOT — reverses the condition
Example: score ≥ 10 AND lives > 0

5 · Building the Mini-Game — Complete Example

The "Run to 10" Mini-Game — Full Technical Specification

This is the minimum viable mini-game that applies all three Module 3 concepts in one programme.

// Setup block — runs once when programme starts
on start {
  set score = 0
  say "Game started! Walk to earn points. Reach 10 to win!"
}

// Event block — fires on every step
on player walk {
  change score by 1
  if score ≥ 10 then {
    say "YOU WIN! Final score: " + score
    fireworks at position player
    set score = 0 // reset for replay
  }
}

Extensions students can add independently (Extra Tasks):

  • Add a lives system: Set lives = 3 at start. If the player falls below Y=60, deduct 1 life. If lives = 0, game over.
  • Add a timer: Use a countdown variable. Decrement it via a repeating event. If timer reaches 0 before score reaches 10, display "Time's Up!"
  • Add power-ups: On Block Broken (Diamond Ore) → change score by +5 as a bonus.
6 · Troubleshooting Catalog — Module 3 (5 Scenarios)
Mistake / SymptomRoot CauseExact FixPrevention Script
Score starts at 1 immediately when /code is run, before the player walks at all "Set score = 0" block is inside the On Player Walk event instead of in a separate On Start or initialisation block Move the "set score = 0" block out of the On Walk event into an On Start block (or directly onto the coding canvas if no start event is used). The On Walk event should only contain "change score by 1" and the conditional check. "Setup code goes in On Start. Game code goes in events. Never mix initialisation with response code."
Variable score never goes above 1 — it resets every walk step instead of accumulating "Set score = 0" is placed inside the On Walk event block, so score is reset to 0 on every step before being incremented to 1 Remove "set score = 0" from inside the On Walk block entirely. Initialisation belongs in On Start only. The On Walk block should only increment: "change score by 1". "Read your code out loud. 'On every walk step: set score to 0, then add 1.' Does that make sense? What number will score always be?"
If–Then condition never triggers, even when score clearly exceeds the target The comparison operator is wrong (using = instead of ≥); OR the variable name in the condition is spelled differently from the variable being incremented (MakeCode is case-sensitive for variable names) Check operator: if the target is score reaching 10 or more, the condition must be score ≥ 10, not score = 10 (which only triggers at exactly 10). Check spelling: "Score" and "score" are different variables in MakeCode. "Use ≥ (greater or equal) for win conditions, not = (exactly equal) — the player might step past the exact number."
Mini-game win condition fires after every step once score reaches 10 — fireworks play repeatedly Score is not being reset to 0 inside the win condition block; after winning, score remains ≥ 10 so every subsequent walk step re-triggers the win Add "set score = 0" as the last line inside the if-then win block. Alternatively, add a boolean "gameActive" variable: set it to false on win, and wrap the entire On Walk event in "if gameActive = true then". "After winning, what should happen to the score? Either reset it to 0 or turn the game off. If you don't reset it, winning triggers forever."
Student cannot understand why "say score" displays a number but they wanted it to display text + number (e.g., "Score: 5") Incorrect block combination: "say" with only the variable block produces the number; to combine text + variable, students must use a "join" block or the string concatenation operator Show students the "text join" block in MakeCode: [say [join ["Score: "] [score]]]. Alternatively, use JavaScript mode to write: say("Score: " + score). Explain: "Say" displays one thing — to show text AND a number, you must join them first into one combined string. "'Say' can only hold one thing. Use 'join' to stick text and numbers together before handing it to Say."
7 · Tutor Pro-Tips & Real-World Analogies

💡 Variables = The "Video Game Score" Analogy (Module 3 Primary Hook)
"Think of every video game you've played. Mario has a coin counter. FIFA has a goal counter. Minecraft has an XP bar. All of those are variables — a box with a name that stores a number, and the number changes as you play. Today, you're going to build the code behind your own score counter."

💡 Events = The "Notification System" Analogy
"Your phone doesn't read every message as soon as it's sent. It waits quietly — and when a message arrives, it notifies you. That notification IS an event. MakeCode event blocks work exactly the same way: they sleep until something happens in Minecraft, then they wake up and run their code."

💡 If–Then = The "Thermostat" Analogy
"A thermostat doesn't heat the house all the time — it checks constantly: 'Is the temperature below 20°C?' If yes, the heater turns on. If no, it stays off. That's exactly what an If–Then block does: it checks a condition and acts only when that condition is true."

💡 The "Block Boundary Visualisation" Trick for Avoiding Outside-Block Errors
Before any student writes code for Module 3, ask them to draw a rectangle on paper. Label it "On Player Walk." Then ask: "Which of your instructions goes INSIDE this rectangle, and which goes outside?" Students who physically draw the boundary before coding make this mistake approximately 70% less often than those who code directly without planning.
