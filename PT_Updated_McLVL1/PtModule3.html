1 · Visão Geral do Módulo — A Transição na Programação

Por que o Módulo 3 é o Ponto de Virada Conceitual

Os Módulos 1 and 2 usaram o MakeCode para automatizar construções físicas — o programa rodava uma vez e produzia um resultado estático. O Módulo 3 introduz a programação reativa e interativa: código que espera algo acontecer no mundo do jogo para então responder. Esse é o paradigma fundamental de praticamente todos os softwares do mundo real, desde aplicativos móveis até motores de jogos. Os alunos não estão mais apenas construindo máquinas — eles estão construindo experiências.

Os três conceitos introduzidos — eventos, variáveis e condicionais — são suficientes para construir qualquer jogo simples. Até o final deste módulo, os alunos terão escrito seu primeiro programa interativo completo.

2 · Programação Orientada a Eventos

O que é um Evento?

Um evento é um sinal de que algo aconteceu no mundo do jogo. Um bloco de evento no MakeCode registra um ouvinte (listener): o código dentro do bloco é executado única e exatamente quando aquele evento dispara. O código não roda continuamente — ele espera e depois responde.

ao caminhar do jogador { ← bloco de evento: dispara a cada passo de caminhada
  alterar score por 1 ← este código roda cada vez que o evento dispara
} ← fora deste bloco = NÃO é ativado pela caminhada

Principais eventos do Módulo 3 que os tutores devem explicar:

  • Ao Caminhar do Jogador: Dispara uma vez por passo dado pelo jogador. Útil para contadores de passos, pontuação baseada em distância, minijogos de caminhada.
  • Ao Quebrar Bloco: Dispara quando o jogador minera/destrói um tipo específico de bloco. Útil para minijogos de coleta de recursos, caça ao tesouro.
  • Ao Comando de Chat [palavra]: Dispara quando o jogador digita a palavra exata no chat. Útil para gatilhos manuais de jogo, códigos de trapaça (cheats), funções de reinício.
  • Ao Pular do Jogador: Dispara uma vez por pulo. Útil para pontuação em parkour, desafios de pulo.
⚠️
Erro Crítico: Código Posicionado Fora do Bloco de Evento
Este é o erro de programação mais comum no Módulo 3. O alunos colocam o bloco "alterar variável por 1" FORA do invólucro do bloco de evento. Quando isso acontece, a alteração da variável é executada uma única vez no carregamento do programa (quando o comando /code roda), e não a cada vez que o evento dispara.
ERRADO — a pontuação muda apenas no início do programa, não ao caminhar:
ao caminhar do jogador { ← bloco de evento vazio
}
alterar score por 1 ← FORA do bloco — roda UMA VEZ e nunca mais
CORRETO — a pontuação muda a cada passo de caminhada:
ao caminhar do jogador {
  alterar score por 1 ← DENTRO do bloco
}
Solução pedagógica: Antes de os alunos programarem, peça para desenharem um retângulo no papel e posicionarem cada instrução dentro ou fora dele. Visualizar fisicamente o limite do bloco previne esse erro.
3 · Variáveis — Contêineres para Dados que Mudam

O que é uma Variável?

Uma variável é um local de armazenamento nomeado na memória que guarda um valor (um número, uma cadeia de texto/string ou um booleano verdadeiro/falso). O valor pode ser lido, gravado e atualizado. No Módulo 3, todas as variáveis serão numéricas — elas guardam pontuações, contadores e cronômetros.

definir score = 0 ← cria a variável e dá a ela um valor inicial
alterar score por +1 ← adiciona 1 a qualquer valor atualmente armazenado
alterar score por -5 ← subtrai 5 do valor atual
definir score = score + 10 ← lê o valor atual, adiciona 10, grava o resultado de volta
say score ← exibe o valor atual no chat (para testes)

Escopo de variáveis no MakeCode: Todas as variáveis são globais no ambiente do MakeCode Education — qualquer bloco de evento pode ler e gravar em qualquer variável. Não existem variáveis locais. Isso significa que, uma vez definida uma variável, todas as partes do programa têm acesso a ela.

Convenção de nomenclatura para ensinar: Use nomes descritivos (score, steps, gems), não letras únicas (x, a). Nomes descritivos tornam a depuração (debugging) possível para crianças de 8 anos — elas conseguem ler o próprio código e entender o que ele faz.

4 · Condicionais — Lógica Se–Então

Blocos Se–Então e Se–Então–Senão

Um condicional permite que o programa tome uma decisão: se uma condição for verdadeira, execute o bloco A; caso contrário, execute o bloco B (a ramificação "senão"). Sem condicionais, o programa não tem capacidade de tomar decisões — ele sempre fará a mesma coisa, independentemente do estado do jogo.

se score ≥ 10 então {
  say "Você Ganhou! Pontuação final: " + score
  fireworks na posição do jogador ← aciona a animação de vitória
  definir score = 0 ← reseta para um novo jogo
}

se score ≥ 10 então { ← variante Se–Então–Senão:
  say "Você Ganhou!"
} senão {
  say "Continue tentando! Faltam " + (10 - score) + " passos"
}

Operadores de comparação que os alunos devem conhecer:

Igualdade & Desigualdade

= igual a · ≠ diferente de
> maior que · < menor que
≥ maior ou igual · ≤ menor ou igual

Lógica Booleana

AND — ambas as condições devem ser verdadeiras
OR — pelo menos uma condição é verdadeira
NOT — inverte a condição
Exemplo: score ≥ 10 AND lives > 0

5 · Construindo o Minijogo — Exemplo Completo

O Minijogo "Corrida até 10" — Especificação Técnica Completa

Este é o minijogo mínimo viável (MVP) que aplica todos os três conceitos do Módulo 3 em um único programa.

// Bloco de configuração — roda uma vez quando o programa inicia
ao iniciar {
  definir score = 0
  say "Jogo iniciado! Caminhe para ganhar pontos. Chegue a 10 para vencer!"
}

// Bloco de evento — dispara a cada passo
ao caminhar do jogador {
  alterar score por 1
  se score ≥ 10 então {
    say "VOCÊ VENCEU! Pontuação final: " + score
    fireworks na posição do jogador
    definir score = 0 // reseta para reiniciar a partida
  }
}

Extensões que os alunos podem adicionar de forma independente (Tarefas Extras):

* **Adicione um sistema de vidas**: Defina `lives = 3` no início. Se o jogador cair abaixo de Y=60, deduza 1 vida. Se lives = 0, fim de jogo. * **Adicione um cronômetro**: Use uma variável de contagem regressiva. Decremente-a através de um evento de repetição. Se o cronômetro chegar a 0 antes da pontuação chegar a 10, exiba "O Tempo Acabou!". * **Adicione power-ups**: Ao Quebrar Bloco (Minério de Diamante) → altere a pontuação por +5 como um bônus.
6 · Catálogo de Resolução de Problemas — Módulo 3 (5 Cenários)
Erro / SintomaCausa RaizSolução ExataRoteiro de Prevenção
A pontuação começa em 1 imediatamente quando o /code é executado, antes mesmo de o jogador caminhar O bloco "definir score = 0" está dentro do evento Ao Caminhar do Jogador em vez de estar em um bloco separado Ao Iniciar ou de inicialização Mova o bloco "definir score = 0" para fora do evento Ao Caminhar e coloque-o em um bloco Ao Iniciar (ou diretamente na tela de código se nenhum evento de início for usado). O evento Ao Caminhar deve conter apenas "alterar score por 1" e a checagem condicional. "O código de configuração vai no Ao Iniciar. O código do jogo vai nos eventos. Nunca misture inicialização com código de resposta."
A variável score nunca passa de 1 — ela reseta a cada passo de caminhada em vez de acumular O bloco "definir score = 0" está posicionado dentro do bloco de evento Ao Caminhar, portanto a pontuação é resetada para 0 a cada passo antes de ser incrementada para 1 Remova completamente o bloco "definir score = 0" de dentro do bloco Ao Caminhar. A inicialização pertence exclusivamente ao bloco Ao Iniciar. O bloco Ao Caminhar deve apenas incrementar: "alterar score por 1". "Leia seu código em voz alta. 'A cada passo de caminhada: defina a pontuação para 0, depois adicione 1.' Isso faz sentido? Qual número a pontuação sempre será?"
A condição Se–Então nunca é acionada, mesmo quando a pontuação claramente ultrapassa a meta O operador de comparação está incorreto (usando = em vez de ≥); OU o nome da variável na condição está escrito de forma diferente da variável que está sendo incrementada (o MakeCode diferencia maiúsculas de minúsculas nos nomes das variáveis) Verifique o operador: se a meta é a pontuação chegar a 10 ou mais, a condição deve ser score ≥ 10, não score = 10 (que só dispara exatamente no 10). Verifique a grafia: "Score" and "score" são variáveis diferentes no MakeCode. "Use ≥ (maior ou igual) para condições de vitória, não = (exatamente igual) — o jogador pode passar direto pelo número exato."
A condição de vitória do minijogo dispara a cada passo assim que a pontuação chega a 10 — os fogos de artifício tocam repetidamente A pontuação não está sendo resetada para 0 dentro do bloco da condição de vitória; após vencer, a pontuação permanece ≥ 10, de modo que cada passo subsequente reativa a condição de vitória Adicione "definir score = 0" como a última linha dentro do bloco se-então de vitória. Alternativamente, introduza uma variável booleana "gameActive": defina-as como falso na vitória e envolva todo o evento Ao Caminhar em "se gameActive = verdadeiro então". "Depois de vencer, o que deve acontecer com a pontuação? Ou resete para 0 ou desligue o jogo. Se você não resetar, a vitória disparará para sempre."
O aluno não entende por que 'say score' exibe apenas um número, sendo que ele queria exibir texto + número (ex: 'Pontuação: 5') Combinação incorreta de blocos: o bloco 'say' com apenas o bloco da variável produz o número; para combinar texto + variável, os alunos devem usar um bloco 'unir texto' (join) ou o operador de concatenação de strings Mostre aos alunos o bloco 'unir texto' (text join) no MakeCode: [say [unir ["Pontuação: "] [score]]]. Como alternativa, use o modo JavaScript para escrever: say("Pontuação: " + score). Explique: O 'Say' exibe uma única coisa — para mostrar texto E um número, você deve uni-los antes em uma string combinada. "O 'Say' só pode segurar uma coisa. Use o bloco 'unir' para colar o texto e os números antes de entregá-los ao Say."
7 · Dicas Pro do Tutor & Analogias do Mundo Real

💡 Variáveis = A Analogia da "Pontuação de Videogame" (Gancho Principal do Módulo 3)
"Pense em cada videogame que você já jogou. O Mario tem um contador de moedas. O FIFA tem um contador de gols. O Minecraft tem uma barra de XP. Todos esses exemplos são variáveis — uma caixa com um nome que guarda um número, e esse número muda conforme você joga. Hoje, você vai construir o código por trás do seu próprio contador de pontuação."

💡 Eventos = A Analogia do "Sistema de Notificações"
"Seu celular não lê cada mensagem no exato momento em que ela é enviada. Ele espera em silêncio — e quando uma mensagem chega, ele te notifica. Essa notificação É um evento. Os blocos de eventos do MakeCode funcionam exatamente da mesma forma: eles dormem até que algo aconteça no Minecraft, então eles acordam e rodam o seu código."

💡 Se–Então = A Analogia do "Termostato"
"Um termostato não aquece a casa o tempo todo — ele checa constantemente: 'A temperatura está abaixo de 20°C?' Se sim, o aquecedor liga. Se não, ele continua desligado. É exatamente isso que um bloco Se–Então faz: ele checa uma condição e age apenas quando essa condição é verdadeira."

💡 O Truque da "Visualização dos Limites do Bloco" para Evitar Erros de Código Fora do Bloco
Antes de qualquer aluno escrever o código para o Módulo 3, peça para ele desenhar um retângulo no papel. Identifique-o como "Ao Caminhar do Jogador". Depois pergunte: "Qual das suas instruções vai DENTRO desse retângulo e qual vai fora?" Alunos que desenham fisicamente o limite antes de programar cometem esse erro aproximadamente 70% menos vezes do que aqueles que codificam diretamente sem planejar.
