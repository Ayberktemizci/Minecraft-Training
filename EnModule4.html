1 · Module Overview — Integration & Escalation

Why Module 4 Is the "Integration Module"

Each previous module introduced isolated concepts. Module 4 is the first module that explicitly asks students to combine every prior skill into a single system:

  • From Module 1: Agent control + Redstone Observer–Piston circuit
  • From Module 2: Coordinate positioning for the farm grid layout
  • From Module 3: Conditional logic ("if the plant is grown, then harvest it")

The result is a fully autonomous farming system that runs without player intervention — the highest complexity build in Level 1. Students who successfully complete Module 4 have demonstrated the ability to design, build, and debug a multi-component automated system.

2 · The Agent as Gardener — Automated Planting

Programming the Agent to Plant in Rows

In Module 1, students moved the Agent manually with basic movement commands. Module 4 uses the Agent to plant seeds automatically in a structured grid — the first use of the Agent as a productive automated tool rather than a demonstration toy.

// Load seeds into Agent inventory first (right-click Agent, add Seeds to slot 1)
on chat command "plant" {
  repeat 5 times { ← plant 5 seeds in a row
    agent place on move down ← places item below as Agent moves
    agent move forward 1 ← advance one block
  }
}

Critical pre-conditions for this to work:

  1. Seeds must be loaded in the Agent's inventory before the chat command fires.
  2. The Agent must be positioned at the first planting spot, facing the correct direction for the row.
  3. The ground must be tilled farmland (use a Hoe on dirt/grass, then water it within 4 blocks).
  4. Agent planting uses "place on move down" which places the item in the block directly below the Agent as it advances.
⚠️
Most common Agent planting failure: The Agent drops seeds on the ground rather than planting them. This happens when the ground is not tilled farmland — seeds placed on regular dirt blocks are treated as dropped items, not planted crops. Always confirm the ground is tilled and within 4 blocks of a water source before running the planting command.

Multi-Row Planting — Nested Loops

For a 5×5 farm, students need nested loops: an outer loop for rows, and an inner loop for the length of each row. This is Module 4's introduction to nested iteration — a concept that directly prefigures 2D array traversal in later programming education.

on chat command "farm" {
  repeat 5 times { ← outer loop: 5 rows
    repeat 5 times { ← inner loop: 5 columns
      agent place on move down
      agent move forward 1
    }
    agent move right 1 ← shift one row to the right
    agent turn around ← face the opposite direction for next row
    agent move back 5 ← return to start of row
  }
}
3 · Conditional Harvesting Circuits

The Observer–Piston Harvest Circuit — Applied to Crops

Students already built an Observer–Piston chain for bamboo in Module 1. Module 4 applies the same principle to wheat, carrots, potatoes, and other farmable crops. The key difference is growth stage detection:

  • Bamboo: grows in distinct whole-block increments — easy to detect.
  • Wheat/Carrots/Potatoes: go through 8 growth stages (0–7). Stage 7 is fully grown. The Observer detects the transition from stage 6 → 7 as a block state change and fires.
  • Repeat the Module 1 circuit: Observer face → crop row. Observer back → Piston input (or Redstone wire). Piston extends horizontally to break the mature crop row.
💡
Teaching the "grow to harvest" conditional: Frame it as an if-then even though it's implemented in hardware: "IF the wheat reaches stage 7 (fully grown) THEN the Observer fires THEN the Piston breaks the crops THEN seeds and food items drop." This language bridges the MakeCode conditional logic from Module 3 to the Redstone conditional logic of Module 4.
4 · Hopper Mechanics & Item Sorting Systems

What Is a Hopper?

A Hopper is a funnel-shaped block with two functions: (1) it collects items that fall into it from above, and (2) it transfers items from its own inventory into an adjacent container (chest, furnace, another Hopper) or deposits them downward through its output spout.

Hopper Placement Rules

  • Output direction: determined by which block the Hopper is facing — its spout points toward the block it was placed onto.
  • To connect to a chest: hold Shift + right-click the chest. Without Shift, the chest inventory opens instead of placing the Hopper.
  • To chain Hoppers: place them end-to-end; items transfer through the chain at 2.5 items per second per Hopper.
  • Capacity: each Hopper holds 5 item stacks (×64 per stack).

Basic Sorting System

A simple two-chest sort: one Hopper above each chest, each loaded with a filter item (the item you want to route to that chest). When items flow through, items matching the filter stack to that Hopper's chest; non-matching items pass to the next Hopper.

Module 4 target: Route wheat to one chest, seeds to another — separating food output from replanting material automatically.

Hopper Transfer Timing — Practical Numbers

Transfer rate: a Hopper processes one item every 4 game ticks (0.2 seconds). At 20 ticks/second, that is 2.5 items per second. For a large farm producing high volumes, a single Hopper chain can become a bottleneck. Solutions:

  • Use parallel Hopper chains — multiple collection points each feeding their own chest.
  • Use a Hopper Minecart over a collection rail — Hopper Minecarts transfer at twice the speed of stationary Hoppers.
5 · The Complete Automation Pipeline

The Four-Stage Pipeline — Detect → Trigger → Collect → Store

Module 4's signature teaching framework. Every fully automated farm system can be described as four sequential stages:

Stage 1: Detect

The Observer block monitors the crop block for a state change (growth stage 6 → 7). When detected, it emits a 1-tick Redstone pulse.

Stage 2: Trigger

The Observer's pulse powers the Piston via Redstone wire. The Piston extends, breaking the fully grown crop block. The crop drops as item entities on the ground.

Stage 3: Collect

The Hopper positioned below the crop row collects falling item entities. Items enter the Hopper's 5-slot inventory automatically within 0.5 seconds of being in Hopper range.

Stage 4: Store

The Hopper transfers collected items into the connected Chest at 2.5 items per second. The sorting system routes different item types to different chests based on Hopper filters.

✅
Teaching moment: Once all four stages are built and tested, have students describe the pipeline in their own words without looking at their code: "What detects? What triggers? What collects? What stores?" This verbal encoding solidifies the conceptual framework and prepares them for Module 5's project design work.
6 · Troubleshooting Catalog — Module 4 (5 Scenarios)
Mistake / SymptomRoot CauseExact FixPrevention Script
Agent drops seeds on the ground rather than planting them The ground beneath the Agent is untilled dirt or grass, not farmland. Seeds placed on non-farmland blocks appear as item entities (dropped items), not planted crops. Before running the plant command: (1) Use a Hoe on each dirt block to till it into farmland. (2) Place a water source within 4 blocks of the farmland to hydrate it (dark brown = hydrated, light tan = dry — crops grow much slower on dry farmland). (3) Confirm by running the Agent over already-tilled farmland first. "Before the Agent can plant, the ground must be tilled and watered. Check the soil colour — dark brown = ready to plant."
Hopper fills up with items but does not transfer them to the chest The Hopper's output spout is not pointing at the chest — it was placed without holding Shift, so the chest inventory opened instead of a connection being formed. The Hopper is now pointing downward into the air. Break the Hopper and replace it: hold Shift, then right-click directly on the chest block. Confirm the Hopper's spout visually connects to the chest side. Alternatively, break the Hopper and re-place it adjacent to the chest in a horizontal orientation (chest can be to the side, not only below). "Always hold Shift when placing a Hopper onto a chest. No Shift = chest opens = Hopper lands wrong."
Observer fires when crops grow but Piston does not extend — crops never harvest The Observer output face is not connected to the Piston. This is the same orientation issue as Module 1 but often more confusing because the farm layout is more complex. Test in isolation: disconnect the Observer from the farm circuit. Place a Redstone Lamp directly behind the Observer output face. If the Lamp lights on crop growth, the Observer works correctly and the problem is downstream (wire gap or Piston orientation). Fix the gap or re-place the Piston facing the correct direction. "Test each stage separately. Observer works? Then check the wire. Wire works? Then check the Piston orientation."
The Agent's nested row-planting loop plants crops in a spiral or random pattern instead of a grid The Agent's facing direction after "turn around" is not correctly oriented — the student used "turn left" twice but forgot that after an odd number of "turn right" commands, "turn left + turn left" ≠ 180° turn in all cases. Or the step count in the return move is wrong. Draw the Agent's path on paper before coding. Mark each turn explicitly. Verify: after completing row 1, does the Agent face the opposite direction? Does "move back 5" return it exactly to the start of the row? Test with 1 row first, observe, then extend to 5 rows. "Draw the path before coding it. Which direction after turn 1? After turn 2? The Agent doesn't know where it 'should' go — only where its code says to go."
The sorting system routes ALL items to the first chest regardless of item type The filter Hopper's inventory slots are not set up correctly — the filter items must partially fill a slot in the Hopper's inventory so that only matching items are held there. Alternatively, no filter was set and the first Hopper takes all items indiscriminately. Hopper sorting requires placing a filter item in each slot of the filter Hopper's inventory, leaving no empty slots. Items that match the filter fill the filter slots; non-matching items pass through. A common simplification: use Locked Hoppers with specific items — this requires a Redstone comparator circuit. For Module 4, use the simpler visual filter method and test with two distinct items (wheat vs. seeds) to confirm routing. "The sorting Hopper needs a 'template' — put one of each target item in its inventory slots. It keeps matching items and lets others pass."
7 · Tutor Pro-Tips & Real-World Analogies

💡 Hopper Sorting = The "Recycling Plant" Analogy
"Think of a recycling plant: different conveyor belts route glass to one bin, paper to another, metal to another. The Hopper sorting system is your digital recycling plant. Wheat goes left, seeds go right — automatically, every time, without anyone manually sorting."

💡 The Four-Stage Pipeline = The "Amazon Warehouse" Analogy
"When you order something on Amazon: (1) A sensor detects your order [Observer]. (2) A robot arm picks the item off the shelf [Piston]. (3) A conveyor belt carries it to the packaging station [Hopper]. (4) It goes into a labelled box in the right section of the warehouse [Chest]. Our farm is the same four stages — just with wheat instead of packages."

💡 Nested Loops = The "Printer Analogy" for 2D Grids
"When a printer prints a page, it moves across one row of dots (inner loop), then drops one line (outer loop), then prints the next row, then drops again. The Agent planting in rows works identically: inner loop = one row of seeds, outer loop = moving to the next row."

💡 Testing in Isolation — The "Debug One Stage at a Time" Rule
For multi-component systems, always test each stage in isolation before integrating. The debugging protocol for Module 4: (1) Test Observer fires on crop growth (use a Lamp). (2) Test Piston extends when powered manually (use a Lever). (3) Test Hopper collects items (throw items above it by hand). (4) Test Hopper routes to correct chest (drop specific items manually). Only when each stage passes individually should students connect the full pipeline.
